Cluster Configuration

One GKE Standard cluster per environment. Production is regional (multi-zone) for high availability.

Node pool strategy
Production: e2-standard-4, 2-5 nodes, on-demand
Staging: e2-standard-2, 1-3 nodes, Spot VMs
Sandbox: e2-standard-2, 1-2 nodes, Spot VMs
Spot VMs save ~70% on non-production. PodDisruptionBudgets handle preemptions gracefully. Production never uses Spot.
Workload Identity
Each service has a dedicated GCP Service Account
Kubernetes SA bound to GCP SA via Workload Identity
No service account keys stored anywhere
Least-privilege IAM per service
Workload Identity eliminates the need for exported JSON keys. Pods authenticate to GCP APIs using their bound identity. This is the GCP-recommended approach and satisfies SOC 2 CC6.1 (logical access).
Autoscaling
HPA: CPU and memory thresholds per deployment
Cluster Autoscaler: scales node pool within min/max bounds
PDBs: minAvailable=1 for all production services
Production maintains 2 minimum replicas per service for zero-downtime deploys. HPA scales pods first, Cluster Autoscaler adds nodes if pod scheduling fails.
Worker Deployment

Same Go binary as the API, started with --mode=worker. Runs as a separate Kubernetes Deployment for independent scaling.

Background jobs (29+ types)
Payout submission and status polling
OFAC/sanctions compliance screening
Webhook delivery to merchants
Batch payout validation and processing
FX quote expiry, exchange, and reconciliation
Rail health checks and balance sync
Settlement and OpenPayD reconciliation
PostgreSQL-backed queue using SELECT FOR UPDATE SKIP LOCKED. Jobs have 3 retry attempts with exponential backoff. Per-job 2-minute timeout. Stale lock cleaner runs every 60s.
Scheduled tasks
Daily 00:00 UTC — Payout summary aggregation
Daily 02:00 UTC — Session expiry checks
Daily 03:00 UTC — Stale account cleanup
Daily 06:00 UTC — Merchant review scheduling
Every 5 min — FX quote expiry, rail health checks
Every 15 min — Rail balance sync, OpenPayD recon
Every 30 min — FX reconciliation
Scheduler uses PostgreSQL advisory locks for leader election. Only one worker pod runs the scheduler at a time, preventing duplicate scheduled jobs across replicas.
Anthos Service Mesh

Istio-based mTLS between all services. No plaintext pod-to-pod traffic.

mTLS enforcement
STRICT mode: all traffic must be mutual TLS
Automatic certificate rotation
Service-to-service identity verification
Satisfies PCI DSS Req 4.1 (encrypt data in transit) and SOC 2 CC6.7 (transmission security). Every inter-service call is encrypted and authenticated.
Network policies
anton-engine accepts traffic from api only
worker accepts no inbound traffic (outbound only)
api, anton-engine, worker all connect to Cloud SQL
api, anton-engine, worker all connect to Redis
Frontends connect to api only
Default deny all ingress per namespace
Network policies enforce least-privilege at the pod level. No service can reach another unless explicitly allowed. Worker has no inbound routes — it only pulls jobs from the PostgreSQL queue and pushes results.

Cloud SQL (PostgreSQL 16)

Shared database between API, Anton Engine, and Worker for zero-latency data access. Private IP only, no public endpoint.

High availability
Production: Regional HA with automatic failover
Staging: Single zone, no HA
Sandbox: Single zone, no HA
Regional HA maintains a synchronous standby in a different zone. Automatic failover with ~30s downtime. Staging and sandbox do not justify the 2x cost.
Backup strategy
Automated daily backups with 14-day retention
Point-in-time recovery (PITR) enabled
Deletion protection enabled in production
Cross-region backup for disaster recovery
PITR allows recovery to any point within the retention window. Satisfies SOC 2 A1.2 (recovery objectives) and ISO 27001 A.12.3 (backup). Deletion protection prevents accidental destruction.
Connection security
Cloud SQL Auth Proxy as sidecar in api, anton-engine, and worker pods
SSL required for all connections
Private IP within VPC, no public endpoint
IAM database authentication via Workload Identity
The Auth Proxy handles SSL and IAM auth transparently. Pods connect to localhost:5432, the proxy tunnels to Cloud SQL over a secure channel. No database credentials stored in secrets.
Memorystore (Redis 7)

Session cache, rate limiting counters, and transient state. Not used for persistent data.

Configuration
Production: Standard tier, 2GB, read replicas, HA
Staging: Basic tier, 1GB, no HA
Sandbox: Basic tier, 1GB, no HA
AUTH + TLS enforced on all tiers
Private IP only, VPC peering
Standard tier provides automatic failover. Basic tier is single-node with no persistence guarantees, acceptable for non-production. Redis data is ephemeral by design.
Basis Theory (Tokenization)

External SaaS vault. The only system that stores raw PII and card data. Core database holds only token references.

Vault architecture
/pii vault — names, addresses, identity documents
/card vault — PANs, CVVs, card data
Per-merchant isolation — one merchant cannot access another's tokens
Client-side SHA-256 fingerprinting for deduplication
Foundational to PCI DSS compliance. Satisfies PCI DSS Req 3.4 (render PAN unreadable) and Req 3.5 (protect cryptographic keys). Raw sensitive data never touches the Anton database.
Access pattern
API tokenizes on ingest, detokenizes on payout submission
Merchant portal reads tokenized display values only
Operations portal can view masked PII for support
Worker detokenizes for rail submission and compliance screening
Detokenization is audited. Every access is logged with who, what, when. BT API keys are stored in Secret Manager and scoped per-service via Workload Identity.

Automatic rotation policies

Each service can only access its own secrets
Satisfies PCI DSS Req 3.5 (protect cryptographic keys) and SOC 2 CC6.1 (logical access). No secrets stored in environment variables or config files.

Repository Structure

Each service is its own Git repo with its own CI pipeline. A separate deploy repo holds all Helm charts and values.

Service repos
api — Go backend, REST API
anton-engine — Go risk/AI engine
worker — Same repo as api (shared binary)
merchant — Next.js merchant portal
operations — Next.js ops portal
www — Next.js marketing site
Each repo has its own GitHub Actions workflow. Pushing to main triggers CI for that service only. Worker shares the api repo but gets its own Helm release.
Deploy repo (GitOps)
Helm charts per service
values-prod.yaml, values-staging.yaml, values-sandbox.yaml
Image tags updated by CI on successful build
ArgoCD watches this repo and auto-syncs
The deploy repo is the single source of truth for what's running in each environment. Every deploy is a Git commit, giving a full audit trail of who deployed what, when.
ArgoCD

GitOps controller running inside each GKE cluster. The only system with cluster credentials.

Configuration
One ArgoCD instance per cluster (prod, staging, sandbox)
Auto-sync enabled with self-heal
Prune orphaned resources on sync
SSO via WorkOS for ArgoCD UI access
Self-heal means if someone manually changes a resource in the cluster, ArgoCD reverts it to match the deploy repo. This prevents configuration drift.
Rollback strategy
Automatic rollback if health checks fail
Manual rollback: revert the image tag commit in deploy repo
ArgoCD UI shows full sync history per application
Database migrations are forward-only (never rolled back)
Rollback is a Git revert, not a kubectl command. Since migrations are forward-only, breaking schema changes must be deployed as multi-step compatible migrations.
Security Boundary

Strict separation between CI (GitHub) and CD (ArgoCD).

Credential isolation
GitHub Actions authenticates to GCP via Workload Identity Federation
GitHub Actions can only push images and update the deploy repo
GitHub Actions has zero cluster access
ArgoCD has cluster-admin scoped to its namespace
No service account keys stored anywhere
If a GitHub Actions workflow is compromised, the attacker can push a malicious image but cannot deploy it — ArgoCD still requires the deploy repo commit, which has branch protection and required reviews. Defense in depth.
Notifications
Deploy success/failure posted to Slack
ArgoCD sync status in Slack channel
CI failures notify via GitHub + Slack
Production deploys ping on-call channel
Slack notifications use ArgoCD Notifications controller. Production deploys are high-visibility — the on-call channel gets notified so the team can watch for issues post-deploy.
Database Migrations
Pre-deploy strategy
Kubernetes Job triggered as a Helm pre-upgrade hook
Runs golang-migrate against Cloud SQL
Forward-only — never edit existing migration files
If migration fails, deploy is aborted
Only runs on api service deploys
Migrations must be backwards-compatible with the running code. This allows the old version to keep serving while the migration runs. Breaking changes require a multi-step deploy: migrate first, deploy new code second.

Cloud Logging

Structured JSON logs from all services. GKE automatically ships container stdout/stderr to Cloud Logging.

Log routing
Application logs: JSON structured (slog in Go, structlog in Python)
GKE system logs: node, kubelet, control plane
Audit logs: Cloud Audit Logs for all GCP API calls
Log sinks to BigQuery for long-term retention
Structured logging is mandatory. No fmt.Println in production. Log sinks to BigQuery satisfy SOC 2 CC7.2 (monitoring) and ISO 27001 A.12.4 (logging) retention requirements.
Log retention
Cloud Logging: 30 days (default)
BigQuery sink: 1 year minimum (compliance)
Audit logs: 400 days (GCP default, non-configurable)
FINTRAC requires 5-year retention for transaction records. BigQuery sink retention will be extended to meet this once FINTRAC registration completes.
Cloud Monitoring

Metrics, uptime checks, and alerting. GKE metrics are automatically collected.

Uptime checks
All public endpoints probed every 5 minutes
Probed from multiple GCP regions simultaneously
HTTPS only — validates SSL certificate health
Endpoints: www, api, merchant, ops (per environment)
If an endpoint fails from 2+ regions, an alert fires immediately. This catches DNS issues, certificate expiry, and full outages before customers report them.
Alert policies
Error rate: alert if 5xx > 1% over 5 minutes
Latency: alert if p99 > 2s over 5 minutes
Pod restarts: alert if > 3 restarts in 10 minutes
Database: CPU > 80%, connections > 80% max
Disk: alert at 80% capacity
Worker: job failure rate > 5% over 15 minutes
Production alerts route to PagerDuty (P1). Staging alerts are Slack-only (informational). Sandbox alerts are disabled.
