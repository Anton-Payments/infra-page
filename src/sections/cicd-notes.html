Branch Strategy

Linear promotion model. Code flows develop to staging to main. Production and sandbox deploy from the same branch.

Branch to environment mapping
develop → Staging
main → Production + Sandbox
Sandbox mirrors production exactly. When code merges to main, both environments deploy simultaneously. This ensures external developers always integrate against the same code running in production.
Pipeline stages
Lint: golangci-lint, eslint, ruff
Test: go test -race, vitest, pytest
Build: Docker multi-stage builds
Push: Artifact Registry (per-environment)
Deploy: Helm upgrade to GKE
All stages must pass before deploy. Docker images are tagged with git SHA for traceability. Artifact Registry is scanned for vulnerabilities before deployment.
Rollback procedure
Helm rollback to previous revision
Previous image always available in Artifact Registry
Database migrations are forward-only (no rollback)
Canary deploys for production (planned)
Rollback is a Helm operation, not a git revert. Since migrations are forward-only, breaking schema changes must be deployed as multi-step compatible migrations. This is a hard rule.
GitHub Actions vs Cloud Build
GitHub Actions: lint, test, build, push image
Cloud Build: optional for GKE deploy step
Workload Identity Federation for keyless auth to GCP
GitHub Actions handles CI. Deploy to GKE can use either GitHub Actions with Workload Identity Federation (no service account keys) or Cloud Build triggered by Artifact Registry push. Both approaches are keyless.
